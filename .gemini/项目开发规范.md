# Alife 项目开发规范

> 基于《.Net官方类库开发准则》、《软件工程》设计思想及实际开发经验整理。
> 规范目标：简单易懂、显著提升可读可维护性、只定有用的规则。

---

## 一、文件结构规范

类成员的排布顺序（前排优先级更高，先满足前置再满足后置）：

1. **按访问修饰符排序**：公开 → 保护 → 私有
2. **按是否静态排序**：静态 → 非静态
3. **按成员类型排序**：数据结构（枚举等）→ 事件 → 字段 → 属性 → 函数
4. **按逻辑关系排序**：成对函数（如 Open/Close）、执行顺序相关等

> **生命周期函数视为私有**：构造函数、`Dispose`、析构函数等由系统自动调用或非功能性的函数，排序时视作私有成员（置于公开功能 API 之后）。
>
> **成对函数紧邻排列**：`Start`/`Stop`、`Open`/`Close` 等成对操作，无论访问级别是否相同，都应相邻放置。


```csharp
public class Door
{
    public enum State { Closing, Opening, Opened }

    public event Action Opened;
    public event Action Closed;

    public State CurrentState => currentState;

    public Task OpenAsync() { ... }
    public Task CloseAsync() { ... }

    State currentState;

    void Initialize() { ... }
}
```

## 二、命名规范

### 基本原则

- 统一采用**英语**命名，禁止拼音
- **私有成员**：小驼峰（camelCase）
- **非私有成员**：大驼峰/帕斯卡（PascalCase）
- 名称**怕短不怕长**，除非是 `html`、`db` 等事实标准，否则不缩写
- 缩写时首字母大写其余小写，如 `DbName`、`HtmlParser`
- 禁止使用下划线（`_`），驼峰命名已能分割词语
- 禁止命名为无意义的 `a`、`b`、`i`（除循环下标外）

### 类型命名规则

| 类型 | 规则 | 示例 |
|------|------|------|
| 接口 | `I` 前缀 | `IRepository` |
| 泛型参数 | `T` 前缀 | `TResult` |
| 函数 | 动词主动语态 | `SendMessage` |
| 事件 | 动词被动语态，时态区分阶段 | `MessageSending`（中）、`MessageSent`（后） |
| 事件处理函数 | `On` + 事件名 | `OnMessageSent` |
| 字段 | 名词/形容词，永远小驼峰 | `isConnected` |
| 属性 | 与对应字段同义，PascalCase | `IsConnected` |
| 布尔型 | 明确表达状态 | `IsDoorOpening` |

### 架构命名约定

| 职责 | 命名后缀 | 说明 |
|------|----------|------|
| 单例实例 | `Instance` | 静态属性 |
| 服务/业务逻辑层（MVP 中的 M） | `+System` / `+Service` | 如 `ChatSystem` |
| 界面层（MVP 中的 V） | `+UI` | 如 `LoginUI` |
| 控制/驱动层（MVP 中的 P） | `+Controller` | 如 `GameController` |

## 三、编码规范

### 字段与属性

- **非常量字段永远私有**，公开访问用属性封装
- 省略默认访问修饰符（如私有的 `private` 可省略），减少冗余

```csharp
// ❌
public int value;
private int count;

// ✓
public int Value { get; private set; }
int count;
```

### 条件判断

- 有多个 if 时用**级联**而非嵌套（满足开闭原则，各块独立）
- 区分**取反**和**判否**：`!` 只用于赋值取反，条件判断要显式 `== false`

```csharp
// ❌ 嵌套
if (a) { if (b) { if (c) { } } }

// ✓ 级联
if (!a) return;
if (!b) return;
DoWork();

// ✓ 判否
if (isConnected == false) return;

// ❌ 判否用 !（容易看漏）
if (!isConnected) return;
```

### var 关键字

避免使用 `var`，应显式写明类型，保持一眼可读。例外：目标类型 `new`（右侧已明确类型）。

```csharp
// ❌
var result = GetUser();

// ✓
User result = GetUser();
List<string> names = new();
```

### 最小作用域

变量、局部函数等应放在最小必要的作用域内，避免污染外层读者视野。

```csharp
// ✓
void Process()
{
    void PreProcess() { ... }
    int step;
    PreProcess();
    ...
}
```

### 现代语法优先

使用能提升可读性的语法糖，让代码更清晰而非炫技：

- `using` 声明（代替 `using` 语句块）
- 模式匹配（`is`、`switch` 表达式）
- 表达式主体（`=>`）
- 目标类型 `new()`
- 空合并运算符（`??`、`?.`）
- 元组解构（`(a, b) = (b, a)`）

> ⚠️ 注意：LINQ 可能产生 GC，性能敏感路径应手写实现。

### 函数设计

- 函数过长时拆分为**局部方法**，保持主函数精简
- 不会失败的操作：不返回 bool/int 状态码
- 可能失败的操作：返回有意义的结果，或抛出异常
- 避免 `return true` 这类永远成功的假象

## 四、注释与文档

- **变量名应自注释**，好的命名优先于注释
- 仅在**复杂逻辑、特殊用法、非直觉行为**时添加注释
- 禁止无效注释（把英文名翻译一遍没有意义）
- 禁止无意义的 XML 文档注释（如 `/// <summary>用户名</summary>` 对着 `UserName` 属性）
- 有价值的注释：解释**为什么**这样做、特殊限制条件、非显而易见的行为

---

## 五、异常与验证

- **内部 API**（调用方可控，我们自己调用）：不做防御，让异常自然暴露，有助于提早发现调用方 bug
- **外部输入**（用户提供的数据，不可控）：应验证格式、做容错处理，给予友好提示
- 禁止"捕获但不处理"（吞掉异常会让排查变成灾难）
- 禁止捕获后仅输出 `e.Message`（丢失堆栈信息，无从定位）
- 禁止为掩盖程序错误而防御性屏蔽（如重复单例静默销毁）

```csharp
// ❌ AI 风格：捕获、判空、吞日志
void Start(Player player)
{
    try {
        if (player == null) return;
        JoinGame(player.Id);
    } catch (Exception e) {
        Log.Error(e.Message);
    }
}

// ✓ 正确：让错误暴露
void Start(Player player)
{
    JoinGame(player.Id);
}
```

## 六、设计原则

### 高内聚低耦合

模块化的标准：**能随拆随删**。任何单个功能应能独立迁移，不牵连无关模块。

### 代码问题自查清单

| 问题 | 解法 |
|------|------|
| 条件分支数量不定，还会持续增加 | 改用委托、抽象类、多态（开闭原则） |
| 某分支在部分场景永远不执行 | 功能存在耦合，应拆分（单一职责） |
| 出现大段复制粘贴 | 提取封装为可复用函数或类 |
| 迁移单个功能会牵连大量无关代码 | 依赖关系有问题，需重构为单向树状结构 |

### 扩展性设计

- 不要预先过度设计（YAGNI 原则）
- 明确需要扩展时才用 `virtual`/`abstract`
- 开关数量不定的分支 → 优先用继承/委托而非 `if`/`switch`

### 层级关系

代码依赖方向应是**单向树状**，不允许越级操作，不允许循环依赖。

参考 DDD 分层（由上到下单向依赖）：

```
Interface（交互层）
    ↓
Application（应用层）—— 组合业务功能
    ↓
Domain（领域层）—— 特定业务通用实现
    ↓
Infrastructure（基础层）—— 跨项目基础服务
```

## 七、性能意识

- 避免明显的性能问题（减少不必要的对象分配，复用结果）
- 不为微优化牺牲可读性
- 性能关键路径才需要极致优化（如避免 LINQ 的 GC、手写排序等）
- 禁止为了日志或调试目的在高频路径留下 `Log` 调用

## 八、AI 辅助编程注意事项

> AI 适合查资料、生成示例，但无法替代工程判断。

使用 AI 生成代码后必须检查：

- [ ] 是否有无意义的判空、try-catch 吞异常？
- [ ] 是否有无效注释（把英文名翻译一遍）？
- [ ] 是否把所有功能堆在一个文件里，没有封装复用？
- [ ] 是否生成了完全不需要的防御代码（掩盖程序错误）？
- [ ] 是否违反了项目命名规范？
- [ ] 代码量是否合理（AI 偏爱写大量冗余代码）？

**原则：AI 是辅助驾驶，不是无人驾驶。最终代码的质量由开发者负责。**

---

## 九、核心思维方式

1. **别没事找事**：只写当前需要的代码，不预测未来需求
2. **专注当前模块**：写一个模块只关心它的输入输出，不同时考虑其他模块
3. **遵循设计原则**：让代码通过自查清单，违反一定有问题
4. **及时重构**：需求变了就改，有隔离的代码修改只烧一个模块
